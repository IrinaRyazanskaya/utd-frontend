# Repository Guidelines

## Структура проекта и модулей

Основная логика реализована на TypeScript и живёт в директории `src/`.

Переиспользуемые React компоненты лежат в `src/components/`, страницы - в `src/pages/`, функции для
работы с API — в `src/api/`, приложение собирается воедино в `src/application/`, вспомогательные
функции пишутся в `src/utils/`.

Клиентская и серверная точки входа расположены в `src/entry-client.tsx` и `src/entry-server.tsx`
соответственно. В приложении настроен SSR, а сервер для SSR запускается из файла `index.js`.

Статические ресурсы и HTML-шаблон хранятся в `public/`. Глобальные стили пишутся в
`src/global-styles.css`.

Для сборки используется Vite. Собранные бандлы складываются в `dist/`.

## Сборка и локальный запуск

Для установки зависимостей необходимо выполнить `npm install`.

Запуск приложения в режиме разработки производится командой `npm run dev`.

Команда `npm run build` компилирует приложение с помощью Vite и сохраняет результат компиляции в
директорию `dist/`. Для запуска production-сборки следует использовать `npm run start`.

Автоматическое форматирование выполняется командой `npm run format`. Команда `npm run lint`
проверяет соответствие кода стандартам стиля, `npm run lint-fix` автоматически исправляет
обнаруженные нарушения, которые поддаются автоматическому исправлению.

## Стиль кода и соглашения об именовании

Файлы именуются в kebab-case (`mail-options.ts`). Для React компонентов создаётся отдельная
директория с именем соответствующим имени компонента - `{component}`. В директории компонента должны
быть следующие файлы: `index.ts` для экспортирования, `{component}.tsx` с разметкой и бизнес
логикой, `{component}.css` для стилей.

В проекте используется ESLint с конфигом `eslint.config.js` и Stylelint с `.stylelintrc.json`.
Форматирование выполняет Prettier: отступ в 2 пробела, двойные кавычки, хвостовые запятые где это
возможно. Длина строки не должна превышать 100 символов.

Переменные и функции именуются в camelCase, типы и интерфейсы – в PascalCase, строковые и числовые
константы в глобальной области видимости - в SCREAMING_SNAKE_CASE.

Предпочтительно использовать ключевое слово `type` вместо `interface` при объявлении типов.

При написании CSS селекторов используется методология БЭМ:

- Блок: `block`
- Элемент: `block__element`
- Булевый модификатор блока: `block_modifier`
- Модификатор блока со значением: `block_modifier_value`
- Булевый модификатор элемента: `block__element_modifier`
- Модификатор элемента со значением: `block__element_modifier_value`

## Тестирование

Модульные и интеграционные тесты отсутствуют. Тестирование производится вручную.

## Коммиты и pull request'ы

Сообщения коммитов пишутся на русском языке в прошедшем времени, например `Починила валидацию`.
Коммиты делаются сфокусированными и описывают одно изменение за раз. В описание PR добавляется
краткое описание задачи и сделанных изменений в коде. Перед публикацией PR необходимо запускать
команды для контроля качества кода: `npm run format-check`, `npm run build-check` и `npm run lint`.
